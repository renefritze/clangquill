# Performance analysis: profiling the benchmark hot paths

A profiler-backed deep dive into the published benchmark results
([`docs/benchmarks.md`](../benchmarks.md), generated by the `benchmark`
workflow from the **v0.2.0 wheel**), run on 2026-07-03. The goal was to
explain the three patterns the published tables show — cold parse much
slower than Doxygen, `incremental` ≈ `cold`, and a flat ~8 s Sphinx stage on
eigen — and to rank improvement opportunities with evidence rather than
intuition.

**Headline: the published numbers no longer describe `main`.** The v0.2.0
wheel predates umbrella-TU batching and the parallel parse driver (its
`ParseOptions` has neither `tu_batch` nor `jobs`). Rebuilt from the current
source, the same machine and inputs give:

| measurement (CLI, `--cache-dir`, this container) | v0.2.0 wheel | current `main` | change |
| --- | --- | --- | --- |
| abseil cold parse (`clangquill build`) | 9.40 s | 1.56 s | **6.0× faster** |
| eigen cold parse | 2.71 s | 1.32 s | 2.1× faster |
| eigen noop | 0.88 s | 0.19 s | 4.6× faster |

The second headline is a correctness finding with large benchmark
implications: **the eigen (and abseil) `clangquill-sphinx` stage crashes
Sphinx mid-build**, so the published 8.2 s "render" number times a build
that dies ~48 % through reading sources — and, because the environment is
never saved, its `noop`/`incremental` numbers can never benefit from
Sphinx's own incrementality. Details in Finding 5 below.

## Method

- Container: Ubuntu 24.04, 4 CPUs, glibc 2.39, Linux 6.18.
- **Reproduction** used the harness as published: `uv sync --frozen
  --project benchmarks` (v0.2.0 wheel, bundled libclang 22, sphinx 9.1.0),
  `python benchmark.py --repos clangquill|abseil|eigen --tools
  clangquill-myst[,clangquill-sphinx]`. Doxygen was unavailable and skipped;
  the clangquill-side *shape* of `docs/benchmarks.md` reproduced within
  ~10 % on all three repos (e.g. abseil myst 9.50/0.42/9.36 s
  cold/noop/incremental vs published 9.36/0.45/9.34 s).
- **Profiling** used a source build (`RelWithDebInfo`, system libclang 18):
  sub-phase wall-clock drivers around the `pipeline.py` seams, `cProfile` on
  the CLI and on `sphinx-build`, `py-spy record --native` (parse stage), and
  `valgrind --tool=callgrind` on `_core.parse_to_sqlite` for instruction-level
  attribution. libclang 18 vs 22 shifts absolute times slightly (and symbol
  counts: 1909 vs 1975 on abseil); relative attribution is what is used below.

## Where the time actually goes (current `main`)

Cold `clangquill build` (parse stage) splits roughly 50/50 between C++ and
Python once the parse is batched and parallel. cProfile of the abseil cold
CLI run (2.0 s under profiler):

| component | time | share |
| --- | --- | --- |
| `_core.parse_to_sqlite` (libclang parse + IR write, C++) | 0.88 s | ~46 % |
| `_rendered_files` (page planning + Jinja render, Python) | 0.83 s | ~44 % |
| — of which Jinja template execution (`render_symbol`) | 0.72 s | |
| — of which `Store` queries (children/symbol/references) | ~0.20 s | |
| cache bookkeeping (`record_parse`, page cache, outputs) | ~0.19 s | |
| interpreter + imports (typer, jinja2, `_core`, metadata) | ~0.15 s | |

Inside the C++ half, callgrind on the clangquill-self parse (2.56 G
instructions) attributes:

| library / function | share of instructions |
| --- | --- |
| libclang 18 (lexing/parsing/sema) | ~64 % |
| CPython (startup + driver) | ~14 % |
| `_core` total | ~11 % |
| — of which `hash::Sha256::transform` | **10.4 % of the whole program** |
| libsqlite3 (IR write) | ~1 % |

Two structural facts frame the findings: `parser.cpp` groups inputs into
umbrella TUs of 64 (`kDefaultTuBatch`) parsed by a thread pool, and
`record_file()` reads + SHA-256-hashes every file each TU transitively
includes, with the dedup set **per batch**, so headers shared by all batches
are re-read and re-hashed once per batch.

## Findings and suggestions, ranked by impact

### Finding 1 — the published benchmark is measuring a superseded architecture

Everything above says the single most effective "optimization" is a
release + re-benchmark: v0.2.0 is 2–6× behind `main` on every clangquill
number in `docs/benchmarks.md`, and the README's "ClangQuill is
single-threaded" fairness note is no longer true (both tools now use all
cores). The `benchmark` workflow only refreshes `docs/benchmarks.md` on
version tags, so the gap will persist until a release.

**Suggested actions:**

- Cut a release (or add a manual-dispatch path that publishes
  `docs/benchmarks.md` from `main`) so the public numbers reflect the
  batching/parallelism work.
- Update the "Benchmarking practices baked in" bullet in
  `benchmarks/README.md` ("Single-threaded & graphviz-free") to describe the
  current all-cores configuration.

### Finding 2 — `incremental` ≈ `cold` is mostly the scenario, partly batch granularity

The per-TU cache works as designed. Measured on abseil (current `main`,
cache warm, `benchmarks/.work/abseil`):

| scenario | time | inputs re-parsed |
| --- | --- | --- |
| noop | 0.16 s | 0 / 61 |
| patch `string_view.h` (the benchmark's patch target) | 1.33 s | 47 / 61 |
| patch a leaf header (`ascii.h`) | 0.81 s | 7 / 61 |
| cold | 1.56 s | 61 / 61 |

The benchmark's fixed patch targets deliberately widely-included headers
(`string_view.h`, `Eigen/Core`'s `Matrix.h`, `symbol.hpp`), which invalidate
77 % of inputs — `incremental` ≈ `cold` is then *correct* invalidation, not
a cache bug. But the leaf-header row shows the remaining inefficiency:
re-parsing 11 % of inputs still costs 52 % of cold, because the 7 stale
inputs are re-parsed as a single umbrella batch on one thread, and every
umbrella re-pays the full parse of the common header prelude
(`parser.cpp:parse_batch`).

**Suggested actions:**

- Use a smaller `tu_batch` for the *incremental* re-parse path
  (`pipeline._parse_tus_into`): stale sets are usually small, so batching
  them at e.g. 8 recovers thread-pool parallelism without changing cold
  behaviour. (Batch composition only affects the re-parsed TUs, so the
  determinism argument for the fixed 64 doesn't bind here.)
- Add a second, leaf-header patch scenario to the benchmark so the report
  shows the incremental win users actually get for local edits, next to the
  worst case.
- Longer term: a preamble/PCH shared *within* the incremental re-parse (the
  stale TUs share the unchanged prelude by construction) would attack the
  "one leaf edit costs half a cold build" floor. Note #90 found a shared PCH
  loses to plain batching for *cold* builds — the incremental path has
  different economics (prelude provably unchanged, small batch).

### Finding 3 — content hashing is the biggest cost clangquill owns in the parse stage

`hash::Sha256::transform` alone is 10.4 % of all instructions in a cold
parse — more than any single libclang function, and ~95 % of `_core`'s own
work. It is a portable scalar SHA-256 (`src/cpp/hash/sha256.cpp`), and the
per-batch `seen` set in `parser.cpp` means shared headers are hashed once
per umbrella TU rather than once per run (abseil: 7 batches, so the common
absl/std prelude is read + hashed ~7×).

The hash only feeds cache invalidation and page fingerprints — nothing
needs a cryptographic digest.

**Suggested actions:**

- Share a process-wide `path → (mtime, size, digest)` cache across batches
  and workers (a mutex-guarded map, or per-worker maps merged like
  `ParsedModule`s) so each file is read and hashed once per run.
- Swap SHA-256 for a fast non-cryptographic content hash (BLAKE3 or XXH3 —
  20–50× faster per byte). Together these turn ~10 % of cold parse into
  noise. (Cache-format version bump required; `CACHE_VERSION` already
  exists for that.)

### Finding 4 — the Python render half: Jinja dominates, plus two cheap wins

The suspected `Store` N+1 pattern is real (eigen: 12,829 SQLite statements
for 1,014 pages, ~12.7/page) but cheap: ~0.14 s execute time, ~10 % of the
render. The bulk is Jinja template execution itself (~0.7 s on abseil,
~1.2 s of 1.4 s profiled on eigen). Within that, cProfile shows two
avoidable costs:

- `Generator._qualify` and `_strip_injected_template_id` call `re.compile`
  with a **per-symbol pattern** (`generator.py:613` and `:630`) — 1,196
  compiles on the eigen render, ~0.15 s, and the `re` module's internal
  cache (512 entries) thrashes on distinct symbol names. A plain
  string-scan replacement (find `spelling` followed by optional `<...>` and
  `(`) or an LRU keyed by spelling removes this.
- `Environment.get_template` runs 13,869 times (0.24 s cumulative) because
  partials are re-looked-up per symbol render; hoisting the handful of
  template objects into the `Generator` (they are immutable per run) trims
  the constant factor.
- The `handle_symbol` forward-decl **linear scan** (`ast_visitor.cpp:169`)
  that looked O(n²) on paper is *not* a problem in practice: callgrind
  shows the whole visitor at <0.5 % of instructions. Not worth an index at
  current scales.

**Suggested actions** (in payoff order): pre-compile/cache the two signature
regexes; cache template lookups; only then consider batching store queries
(e.g. one `children` query per page instead of per symbol) — the profile
says it's a ~10 % render item, not the headline.

### Finding 5 — the Sphinx stage on eigen/abseil crashes, and the crash shapes all three published render numbers

`clangquill-sphinx` exits with code 2 on eigen and abseil in the published
run (footnoted as "diagnostics in logs"), but this is not a diagnostic-level
failure: **Sphinx aborts with an unhandled `AttributeError` in the C++
domain** (`sphinx/domains/cpp/__init__.py:440`,
`_object_hierarchy_parts: 'NoneType' object has no attribute
'get_full_nested_name'`) about 48 % through the read phase.

Reproduced and bisected on eigen: the trigger sits in `api/Eigen.md`, where
degraded extraction (eigen without its full include tree) emits template
functions as colliding variable declarations — e.g. `{cpp:var} int
Eigen::internal::plogical_shift_right` next to other declarations of the
same name. Sphinx's cpp domain mishandles the duplicate and crashes instead
of warning.

Consequences for the published numbers:

- The 8.2 s eigen "cold" render is **half a build** — with the crashing
  page excluded, a *successful* cold `sphinx-build -j auto` takes **58.7 s**.
- Because the crash prevents the environment pickle from being written,
  every scenario re-reads everything: that is why the published eigen
  sphinx row is flat (8.18/8.13/8.21 s). With the crash out of the way,
  **noop drops to 1.6 s** (a 36× cold→noop speedup — Sphinx's own
  incrementality works fine, as the clangquill-self row already showed:
  1.79 s cold / 0.62 s noop, exit 0).

**Suggested actions:**

- In the generator, detect and de-duplicate same-name declarations of
  conflicting kinds on a page (or emit the degraded ones as plain code
  blocks instead of `cpp:*` directives) so a mis-extracted symbol can never
  take down the whole Sphinx build. Also report the crash upstream — Sphinx
  should warn, not raise.
- In the harness, treat a non-zero `sphinx-build` exit as an **invalid
  sample** (report "failed" instead of a time). A crashed build's wall
  clock is not a render time, and publishing it both understates cold and
  hides the noop win.

### Finding 6 — one page holds 84 % of eigen's output

With `group_by="symbol"` (the benchmark default), a single root namespace
collapses the whole subtree onto one page: `api/Eigen.md` is 765 KB of the
910 KB total. That page is simultaneously the slowest to render, the
slowest for myst/Sphinx to parse, and the page the crash lives on. The
work metric "1030 pages" hides that 1029 of them are macro stubs.

**Suggested action:** benchmark (and recommend, for namespace-rooted
libraries) `group_by="namespace"` or `"class"`; it splits the monolith into
balanced pages, which parallelizes the Sphinx read phase and avoids
re-rendering one giant page on every symbol change. This likely also
shrinks the real 58.7 s cold Sphinx cost: the duplicate-declaration
warnings that flood that build are concentrated in the monolithic page.

### Smaller observations

- **Fixed CLI overhead** is ~0.15 s (imports: typer, jinja2,
  `importlib.metadata`, `_core`) — one third of a warm noop build of a
  small project. Lazy-importing the generator/Jinja stack for the
  cache-hit path would make noop builds feel instant.
- **`fstatat64` is ~16 %** of py-spy's native samples during the abseil
  parse — libclang stat-ing headers across 7 umbrella TUs. A shared
  `CXIndex`-level VFS/stat cache isn't exposed by libclang's C API, so this
  is mostly a ceiling on batching gains, not an action item.
- The benchmark's own methodology held up well under re-measurement:
  medians reproduced within ~10 % in a shared cloud container, and the
  work-metric columns made the extraction differences (1909 vs 1975
  symbols under libclang 18 vs 22) visible immediately.

## Threats to validity

- Profiling ran against system libclang 18 (the wheel bundles 22); libclang
  shares and symbol counts differ a few percent between the two.
- cProfile inflates Python-heavy sections (~2× on the eigen render);
  shares quoted together always come from the same profile.
- The container is a shared 4-CPU cloud instance without a pinned governor;
  all comparisons above are within-machine, same-run-conditions.
- Callgrind numbers come from the clangquill-self repo (single umbrella
  batch); the cross-batch hash-duplication multiplier on abseil/eigen is
  inferred from `parser.cpp` structure, not measured directly.

## Reproduction quick reference

```bash
# published-wheel timings
uv sync --frozen --project benchmarks
uv run --project benchmarks python benchmarks/benchmark.py \
    --repos abseil --tools clangquill-myst --repeat 2 --warmup 1

# source build for profiling
uv venv .venv-src --python 3.13
SKBUILD_CMAKE_BUILD_TYPE=RelWithDebInfo uv pip install --python .venv-src/bin/python .

# C++ attribution (callgrind), render attribution (cProfile)
valgrind --tool=callgrind .venv-src/bin/python -c '<parse-only driver>'
.venv-src/bin/python -m cProfile -o out.prof .venv-src/bin/clangquill build ...
```
